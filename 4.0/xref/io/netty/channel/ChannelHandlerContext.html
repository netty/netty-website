31  /**
32   * Enables a {@link ChannelHandler} to interact with its {@link ChannelPipeline}
33   * and other handlers.  A handler can notify the next {@link ChannelHandler} in the {@link ChannelPipeline},
34   * modify the {@link ChannelPipeline} it belongs to dynamically.
35   *
36   * <h3>Notify</h3>
37   *
38   * You can notify the closest handler in the
39   * same {@link ChannelPipeline} by calling one of the various methods provided here.
40   * Please refer to {@link ChannelPipeline} to understand how an event flows.
41   *
42   * <h3>Modifying a pipeline</h3>
43   *
44   * You can get the {@link ChannelPipeline} your handler belongs to by calling
45   * {@link #pipeline()}.  A non-trivial application could insert, remove, or
46   * replace handlers in the pipeline dynamically at runtime.
47   *
48   * <h3>Retrieving for later use</h3>
49   *
50   * You can keep the {@link ChannelHandlerContext} for later use, such as
51   * triggering an event outside the handler methods, even from a different thread.
52   * <pre>
53   * public class MyHandler extends {@link ChannelDuplexHandler} {
54   *
55   *     <b>private {@link ChannelHandlerContext} ctx;</b>
56   *
57   *     public void beforeAdd({@link ChannelHandlerContext} ctx) {
58   *         <b>this.ctx = ctx;</b>
59   *     }
60   *
61   *     public void login(String username, password) {
62   *         ctx.write(new LoginMessage(username, password));
63   *     }
64   *     ...
65   * }
66   * </pre>
67   *
68   * <h3>Storing stateful information</h3>
69   *
70   * {@link #attr(AttributeKey)} allow you to
71   * store and access stateful information that is related with a handler and its
72   * context.  Please refer to {@link ChannelHandler} to learn various recommended
73   * ways to manage stateful information.
74   *
75   * <h3>A handler can have more than one context</h3>
76   *
77   * Please note that a {@link ChannelHandler} instance can be added to more than
78   * one {@link ChannelPipeline}.  It means a single {@link ChannelHandler}
79   * instance can have more than one {@link ChannelHandlerContext} and therefore
80   * the single instance can be invoked with different
81   * {@link ChannelHandlerContext}s if it is added to one or more
82   * {@link ChannelPipeline}s more than once.
83   * <p>
84   * For example, the following handler will have as many independent {@link AttributeKey}s
85   * as how many times it is added to pipelines, regardless if it is added to the
86   * same pipeline multiple times or added to different pipelines multiple times:
87   * <pre>
88   * public class FactorialHandler extends {@link ChannelInboundHandlerAdapter}&lt{@link Integer}&gt {
89   *
90   *   private final {@link AttributeKey}&lt{@link Integer}&gt counter =
91   *           new {@link AttributeKey}&lt{@link Integer}&gt("counter");
92   *
93   *   // This handler will receive a sequence of increasing integers starting
94   *   // from 1.
95   *   {@code @Override}
96   *   public void channelRead({@link ChannelHandlerContext} ctx, {@link Integer} integer) {
97   *     {@link Attribute}&lt{@link Integer}&gt attr = ctx.getAttr(counter);
98   *     Integer a = ctx.getAttr(counter).get();
99   *
100  *     if (a == null) {
101  *       a = 1;
102  *     }
103  *
104  *     attr.set(a * integer));
105  *   }
106  * }
107  *
108  * // Different context objects are given to "f1", "f2", "f3", and "f4" even if
109  * // they refer to the same handler instance.  Because the FactorialHandler
110  * // stores its state in a context object (as an (using an {@link AttributeKey}), the factorial is
111  * // calculated correctly 4 times once the two pipelines (p1 and p2) are active.
112  * FactorialHandler fh = new FactorialHandler();
113  *
114  * {@link ChannelPipeline} p1 = {@link Channels}.pipeline();
115  * p1.addLast("f1", fh);
116  * p1.addLast("f2", fh);
117  *
118  * {@link ChannelPipeline} p2 = {@link Channels}.pipeline();
119  * p2.addLast("f3", fh);
120  * p2.addLast("f4", fh);
121  * </pre>
122  *
123  * <h3>Additional resources worth reading</h3>
124  * <p>
125  * Please refer to the {@link ChannelHandler}, and
126  * {@link ChannelPipeline} to find out more about inbound and outbound operations,
127  * what fundamental differences they have, how they flow in a  pipeline,  and how to handle
128  * the operation in your application.
129  */
