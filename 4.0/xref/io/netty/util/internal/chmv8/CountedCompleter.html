25  /**
26   * A {@link ForkJoinTask} with a completion action performed when
27   * triggered and there are no remaining pending
28   * actions. CountedCompleters are in general more robust in the
29   * presence of subtask stalls and blockage than are other forms of
30   * ForkJoinTasks, but are less intuitive to program.  Uses of
31   * CountedCompleter are similar to those of other completion based
32   * components (such as {@link java.nio.channels.CompletionHandler})
33   * except that multiple <em>pending</em> completions may be necessary
34   * to trigger the {@link #onCompletion} action, not just one. Unless
35   * initialized otherwise, the {@link #getPendingCount pending count}
36   * starts at zero, but may be (atomically) changed using methods
37   * {@link #setPendingCount}, {@link #addToPendingCount}, and {@link
38   * #compareAndSetPendingCount}. Upon invocation of {@link
39   * #tryComplete}, if the pending action count is nonzero, it is
40   * decremented; otherwise, the completion action is performed, and if
41   * this completer itself has a completer, the process is continued
42   * with its completer.  As is the case with related synchronization
43   * components such as {@link java.util.concurrent.Phaser Phaser} and
44   * {@link java.util.concurrent.Semaphore Semaphore}, these methods
45   * affect only internal counts; they do not establish any further
46   * internal bookkeeping. In particular, the identities of pending
47   * tasks are not maintained. As illustrated below, you can create
48   * subclasses that do record some or all pending tasks or their
49   * results when needed.  As illustrated below, utility methods
50   * supporting customization of completion traversals are also
51   * provided. However, because CountedCompleters provide only basic
52   * synchronization mechanisms, it may be useful to create further
53   * abstract subclasses that maintain linkages, fields, and additional
54   * support methods appropriate for a set of related usages.
55   *
56   * <p>A concrete CountedCompleter class must define method {@link
57   * #compute}, that should in most cases (as illustrated below), invoke
58   * {@code tryComplete()} once before returning. The class may also
59   * optionally override method {@link #onCompletion} to perform an
60   * action upon normal completion, and method {@link
61   * #onExceptionalCompletion} to perform an action upon any exception.
62   *
63   * <p>CountedCompleters most often do not bear results, in which case
64   * they are normally declared as {@code CountedCompleter<Void>}, and
65   * will always return {@code null} as a result value.  In other cases,
66   * you should override method {@link #getRawResult} to provide a
67   * result from {@code join(), invoke()}, and related methods.  In
68   * general, this method should return the value of a field (or a
69   * function of one or more fields) of the CountedCompleter object that
70   * holds the result upon completion. Method {@link #setRawResult} by
71   * default plays no role in CountedCompleters.  It is possible, but
72   * rarely applicable, to override this method to maintain other
73   * objects or fields holding result data.
74   *
75   * <p>A CountedCompleter that does not itself have a completer (i.e.,
76   * one for which {@link #getCompleter} returns {@code null}) can be
77   * used as a regular ForkJoinTask with this added functionality.
78   * However, any completer that in turn has another completer serves
79   * only as an internal helper for other computations, so its own task
80   * status (as reported in methods such as {@link ForkJoinTask#isDone})
81   * is arbitrary; this status changes only upon explicit invocations of
82   * {@link #complete}, {@link ForkJoinTask#cancel}, {@link
83   * ForkJoinTask#completeExceptionally} or upon exceptional completion
84   * of method {@code compute}. Upon any exceptional completion, the
85   * exception may be relayed to a task's completer (and its completer,
86   * and so on), if one exists and it has not otherwise already
87   * completed. Similarly, cancelling an internal CountedCompleter has
88   * only a local effect on that completer, so is not often useful.
89   *
90   * <p><b>Sample Usages.</b>
91   *
92   * <p><b>Parallel recursive decomposition.</b> CountedCompleters may
93   * be arranged in trees similar to those often used with {@link
94   * RecursiveAction}s, although the constructions involved in setting
95   * them up typically vary. Here, the completer of each task is its
96   * parent in the computation tree. Even though they entail a bit more
97   * bookkeeping, CountedCompleters may be better choices when applying
98   * a possibly time-consuming operation (that cannot be further
99   * subdivided) to each element of an array or collection; especially
100  * when the operation takes a significantly different amount of time
101  * to complete for some elements than others, either because of
102  * intrinsic variation (for example I/O) or auxiliary effects such as
103  * garbage collection.  Because CountedCompleters provide their own
104  * continuations, other threads need not block waiting to perform
105  * them.
106  *
107  * <p>For example, here is an initial version of a class that uses
108  * divide-by-two recursive decomposition to divide work into single
109  * pieces (leaf tasks). Even when work is split into individual calls,
110  * tree-based techniques are usually preferable to directly forking
111  * leaf tasks, because they reduce inter-thread communication and
112  * improve load balancing. In the recursive case, the second of each
113  * pair of subtasks to finish triggers completion of its parent
114  * (because no result combination is performed, the default no-op
115  * implementation of method {@code onCompletion} is not overridden). A
116  * static utility method sets up the base task and invokes it
117  * (here, implicitly using the {@link ForkJoinPool#commonPool()}).
118  *
119  * <pre> {@code
120  * class MyOperation<E> { void apply(E e) { ... }  }
121  *
122  * class ForEach<E> extends CountedCompleter<Void> {
123  *
124  *   public static <E> void forEach(E[] array, MyOperation<E> op) {
125  *     new ForEach<E>(null, array, op, 0, array.length).invoke();
126  *   }
127  *
128  *   final E[] array; final MyOperation<E> op; final int lo, hi;
129  *   ForEach(CountedCompleter<?> p, E[] array, MyOperation<E> op, int lo, int hi) {
130  *     super(p);
131  *     this.array = array; this.op = op; this.lo = lo; this.hi = hi;
132  *   }
133  *
134  *   public void compute() { // version 1
135  *     if (hi - lo >= 2) {
136  *       int mid = (lo + hi) >>> 1;
137  *       setPendingCount(2); // must set pending count before fork
138  *       new ForEach(this, array, op, mid, hi).fork(); // right child
139  *       new ForEach(this, array, op, lo, mid).fork(); // left child
140  *     }
141  *     else if (hi > lo)
142  *       op.apply(array[lo]);
143  *     tryComplete();
144  *   }
145  * }}</pre>
146  *
147  * This design can be improved by noticing that in the recursive case,
148  * the task has nothing to do after forking its right task, so can
149  * directly invoke its left task before returning. (This is an analog
150  * of tail recursion removal.)  Also, because the task returns upon
151  * executing its left task (rather than falling through to invoke
152  * {@code tryComplete}) the pending count is set to one:
153  *
154  * <pre> {@code
155  * class ForEach<E> ...
156  *   public void compute() { // version 2
157  *     if (hi - lo >= 2) {
158  *       int mid = (lo + hi) >>> 1;
159  *       setPendingCount(1); // only one pending
160  *       new ForEach(this, array, op, mid, hi).fork(); // right child
161  *       new ForEach(this, array, op, lo, mid).compute(); // direct invoke
162  *     }
163  *     else {
164  *       if (hi > lo)
165  *         op.apply(array[lo]);
166  *       tryComplete();
167  *     }
168  *   }
169  * }</pre>
170  *
171  * As a further improvement, notice that the left task need not even
172  * exist.  Instead of creating a new one, we can iterate using the
173  * original task, and add a pending count for each fork. Additionally,
174  * because no task in this tree implements an {@link #onCompletion}
175  * method, {@code tryComplete()} can be replaced with {@link
176  * #propagateCompletion}.
177  *
178  * <pre> {@code
179  * class ForEach<E> ...
180  *   public void compute() { // version 3
181  *     int l = lo,  h = hi;
182  *     while (h - l >= 2) {
183  *       int mid = (l + h) >>> 1;
184  *       addToPendingCount(1);
185  *       new ForEach(this, array, op, mid, h).fork(); // right child
186  *       h = mid;
187  *     }
188  *     if (h > l)
189  *       op.apply(array[l]);
190  *     propagateCompletion();
191  *   }
192  * }</pre>
193  *
194  * Additional improvements of such classes might entail precomputing
195  * pending counts so that they can be established in constructors,
196  * specializing classes for leaf steps, subdividing by say, four,
197  * instead of two per iteration, and using an adaptive threshold
198  * instead of always subdividing down to single elements.
199  *
200  * <p><b>Searching.</b> A tree of CountedCompleters can search for a
201  * value or property in different parts of a data structure, and
202  * report a result in an {@link
203  * java.util.concurrent.atomic.AtomicReference AtomicReference} as
204  * soon as one is found. The others can poll the result to avoid
205  * unnecessary work. (You could additionally {@linkplain #cancel
206  * cancel} other tasks, but it is usually simpler and more efficient
207  * to just let them notice that the result is set and if so skip
208  * further processing.)  Illustrating again with an array using full
209  * partitioning (again, in practice, leaf tasks will almost always
210  * process more than one element):
211  *
212  * <pre> {@code
213  * class Searcher<E> extends CountedCompleter<E> {
214  *   final E[] array; final AtomicReference<E> result; final int lo, hi;
215  *   Searcher(CountedCompleter<?> p, E[] array, AtomicReference<E> result, int lo, int hi) {
216  *     super(p);
217  *     this.array = array; this.result = result; this.lo = lo; this.hi = hi;
218  *   }
219  *   public E getRawResult() { return result.get(); }
220  *   public void compute() { // similar to ForEach version 3
221  *     int l = lo,  h = hi;
222  *     while (result.get() == null && h >= l) {
223  *       if (h - l >= 2) {
224  *         int mid = (l + h) >>> 1;
225  *         addToPendingCount(1);
226  *         new Searcher(this, array, result, mid, h).fork();
227  *         h = mid;
228  *       }
229  *       else {
230  *         E x = array[l];
231  *         if (matches(x) && result.compareAndSet(null, x))
232  *           quietlyCompleteRoot(); // root task is now joinable
233  *         break;
234  *       }
235  *     }
236  *     tryComplete(); // normally complete whether or not found
237  *   }
238  *   boolean matches(E e) { ... } // return true if found
239  *
240  *   public static <E> E search(E[] array) {
241  *       return new Searcher<E>(null, array, new AtomicReference<E>(), 0, array.length).invoke();
242  *   }
243  * }}</pre>
244  *
245  * In this example, as well as others in which tasks have no other
246  * effects except to compareAndSet a common result, the trailing
247  * unconditional invocation of {@code tryComplete} could be made
248  * conditional ({@code if (result.get() == null) tryComplete();})
249  * because no further bookkeeping is required to manage completions
250  * once the root task completes.
251  *
252  * <p><b>Recording subtasks.</b> CountedCompleter tasks that combine
253  * results of multiple subtasks usually need to access these results
254  * in method {@link #onCompletion}. As illustrated in the following
255  * class (that performs a simplified form of map-reduce where mappings
256  * and reductions are all of type {@code E}), one way to do this in
257  * divide and conquer designs is to have each subtask record its
258  * sibling, so that it can be accessed in method {@code onCompletion}.
259  * This technique applies to reductions in which the order of
260  * combining left and right results does not matter; ordered
261  * reductions require explicit left/right designations.  Variants of
262  * other streamlinings seen in the above examples may also apply.
263  *
264  * <pre> {@code
265  * class MyMapper<E> { E apply(E v) {  ...  } }
266  * class MyReducer<E> { E apply(E x, E y) {  ...  } }
267  * class MapReducer<E> extends CountedCompleter<E> {
268  *   final E[] array; final MyMapper<E> mapper;
269  *   final MyReducer<E> reducer; final int lo, hi;
270  *   MapReducer<E> sibling;
271  *   E result;
272  *   MapReducer(CountedCompleter<?> p, E[] array, MyMapper<E> mapper,
273  *              MyReducer<E> reducer, int lo, int hi) {
274  *     super(p);
275  *     this.array = array; this.mapper = mapper;
276  *     this.reducer = reducer; this.lo = lo; this.hi = hi;
277  *   }
278  *   public void compute() {
279  *     if (hi - lo >= 2) {
280  *       int mid = (lo + hi) >>> 1;
281  *       MapReducer<E> left = new MapReducer(this, array, mapper, reducer, lo, mid);
282  *       MapReducer<E> right = new MapReducer(this, array, mapper, reducer, mid, hi);
283  *       left.sibling = right;
284  *       right.sibling = left;
285  *       setPendingCount(1); // only right is pending
286  *       right.fork();
287  *       left.compute();     // directly execute left
288  *     }
289  *     else {
290  *       if (hi > lo)
291  *           result = mapper.apply(array[lo]);
292  *       tryComplete();
293  *     }
294  *   }
295  *   public void onCompletion(CountedCompleter<?> caller) {
296  *     if (caller != this) {
297  *       MapReducer<E> child = (MapReducer<E>)caller;
298  *       MapReducer<E> sib = child.sibling;
299  *       if (sib == null || sib.result == null)
300  *         result = child.result;
301  *       else
302  *         result = reducer.apply(child.result, sib.result);
303  *     }
304  *   }
305  *   public E getRawResult() { return result; }
306  *
307  *   public static <E> E mapReduce(E[] array, MyMapper<E> mapper, MyReducer<E> reducer) {
308  *     return new MapReducer<E>(null, array, mapper, reducer,
309  *                              0, array.length).invoke();
310  *   }
311  * }}</pre>
312  *
313  * Here, method {@code onCompletion} takes a form common to many
314  * completion designs that combine results. This callback-style method
315  * is triggered once per task, in either of the two different contexts
316  * in which the pending count is, or becomes, zero: (1) by a task
317  * itself, if its pending count is zero upon invocation of {@code
318  * tryComplete}, or (2) by any of its subtasks when they complete and
319  * decrement the pending count to zero. The {@code caller} argument
320  * distinguishes cases.  Most often, when the caller is {@code this},
321  * no action is necessary. Otherwise the caller argument can be used
322  * (usually via a cast) to supply a value (and/or links to other
323  * values) to be combined.  Assuming proper use of pending counts, the
324  * actions inside {@code onCompletion} occur (once) upon completion of
325  * a task and its subtasks. No additional synchronization is required
326  * within this method to ensure thread safety of accesses to fields of
327  * this task or other completed tasks.
328  *
329  * <p><b>Completion Traversals</b>. If using {@code onCompletion} to
330  * process completions is inapplicable or inconvenient, you can use
331  * methods {@link #firstComplete} and {@link #nextComplete} to create
332  * custom traversals.  For example, to define a MapReducer that only
333  * splits out right-hand tasks in the form of the third ForEach
334  * example, the completions must cooperatively reduce along
335  * unexhausted subtask links, which can be done as follows:
336  *
337  * <pre> {@code
338  * class MapReducer<E> extends CountedCompleter<E> { // version 2
339  *   final E[] array; final MyMapper<E> mapper;
340  *   final MyReducer<E> reducer; final int lo, hi;
341  *   MapReducer<E> forks, next; // record subtask forks in list
342  *   E result;
343  *   MapReducer(CountedCompleter<?> p, E[] array, MyMapper<E> mapper,
344  *              MyReducer<E> reducer, int lo, int hi, MapReducer<E> next) {
345  *     super(p);
346  *     this.array = array; this.mapper = mapper;
347  *     this.reducer = reducer; this.lo = lo; this.hi = hi;
348  *     this.next = next;
349  *   }
350  *   public void compute() {
351  *     int l = lo,  h = hi;
352  *     while (h - l >= 2) {
353  *       int mid = (l + h) >>> 1;
354  *       addToPendingCount(1);
355  *       (forks = new MapReducer(this, array, mapper, reducer, mid, h, forks)).fork;
356  *       h = mid;
357  *     }
358  *     if (h > l)
359  *       result = mapper.apply(array[l]);
360  *     // process completions by reducing along and advancing subtask links
361  *     for (CountedCompleter<?> c = firstComplete(); c != null; c = c.nextComplete()) {
362  *       for (MapReducer t = (MapReducer)c, s = t.forks;  s != null; s = t.forks = s.next)
363  *         t.result = reducer.apply(t.result, s.result);
364  *     }
365  *   }
366  *   public E getRawResult() { return result; }
367  *
368  *   public static <E> E mapReduce(E[] array, MyMapper<E> mapper, MyReducer<E> reducer) {
369  *     return new MapReducer<E>(null, array, mapper, reducer,
370  *                              0, array.length, null).invoke();
371  *   }
372  * }}</pre>
373  *
374  * <p><b>Triggers.</b> Some CountedCompleters are themselves never
375  * forked, but instead serve as bits of plumbing in other designs;
376  * including those in which the completion of one of more async tasks
377  * triggers another async task. For example:
378  *
379  * <pre> {@code
380  * class HeaderBuilder extends CountedCompleter<...> { ... }
381  * class BodyBuilder extends CountedCompleter<...> { ... }
382  * class PacketSender extends CountedCompleter<...> {
383  *   PacketSender(...) { super(null, 1); ... } // trigger on second completion
384  *   public void compute() { } // never called
385  *   public void onCompletion(CountedCompleter<?> caller) { sendPacket(); }
386  * }
387  * // sample use:
388  * PacketSender p = new PacketSender();
389  * new HeaderBuilder(p, ...).fork();
390  * new BodyBuilder(p, ...).fork();
391  * }</pre>
392  *
393  * @since 1.8
394  * @author Doug Lea
395  */
396 @SuppressWarnings("all")
397 abstract class CountedCompleter<T> extends ForkJoinTask<T> {
398     private static final long serialVersionUID = 5232453752276485070L;
399 
400     /** This task's completer, or null if none */
401     final CountedCompleter<?> completer;
402     /** The number of pending tasks until completion */
403     volatile int pending;
404 
405     /**
406      * Creates a new CountedCompleter with the given completer
407      * and initial pending count.
408      *
409      * @param completer this task's completer, or {@code null} if none
410      * @param initialPendingCount the initial pending count
411      */
412     protected CountedCompleter(CountedCompleter<?> completer,
413                                int initialPendingCount) {
414         this.completer = completer;
415         this.pending = initialPendingCount;
416     }
417 
418     /**
419      * Creates a new CountedCompleter with the given completer
420      * and an initial pending count of zero.
421      *
422      * @param completer this task's completer, or {@code null} if none
423      */
424     protected CountedCompleter(CountedCompleter<?> completer) {
425         this.completer = completer;
426     }
427 
428     /**
429      * Creates a new CountedCompleter with no completer
430      * and an initial pending count of zero.
431      */
432     protected CountedCompleter() {
433         this.completer = null;
434     }
435 
436     /**
437      * The main computation performed by this task.
438      */
439     public abstract void compute();
440 
441     /**
442      * Performs an action when method {@link #tryComplete} is invoked
443      * and the pending count is zero, or when the unconditional
444      * method {@link #complete} is invoked.  By default, this method
445      * does nothing. You can distinguish cases by checking the
446      * identity of the given caller argument. If not equal to {@code
447      * this}, then it is typically a subtask that may contain results
448      * (and/or links to other results) to combine.
449      *
450      * @param caller the task invoking this method (which may
451      * be this task itself)
452      */
453     public void onCompletion(CountedCompleter<?> caller) {
454     }
455 
456     /**
457      * Performs an action when method {@link #completeExceptionally}
458      * is invoked or method {@link #compute} throws an exception, and
459      * this task has not otherwise already completed normally. On
460      * entry to this method, this task {@link
461      * ForkJoinTask#isCompletedAbnormally}.  The return value of this
462      * method controls further propagation: If {@code true} and this
463      * task has a completer, then this completer is also completed
464      * exceptionally.  The default implementation of this method does
465      * nothing except return {@code true}.
466      *
467      * @param ex the exception
468      * @param caller the task invoking this method (which may
469      * be this task itself)
470      * @return true if this exception should be propagated to this
471      * task's completer, if one exists
472      */
473     public boolean onExceptionalCompletion(Throwable ex, CountedCompleter<?> caller) {
474         return true;
475     }
476 
477     /**
478      * Returns the completer established in this task's constructor,
479      * or {@code null} if none.
480      *
481      * @return the completer
482      */
483     public final CountedCompleter<?> getCompleter() {
484         return completer;
485     }
486 
487     /**
488      * Returns the current pending count.
489      *
490      * @return the current pending count
491      */
492     public final int getPendingCount() {
493         return pending;
494     }
495 
496     /**
497      * Sets the pending count to the given value.
498      *
499      * @param count the count
500      */
501     public final void setPendingCount(int count) {
502         pending = count;
503     }
504 
505     /**
506      * Adds (atomically) the given value to the pending count.
507      *
508      * @param delta the value to add
509      */
510     public final void addToPendingCount(int delta) {
511         int c; // note: can replace with intrinsic in jdk8
512         do {} while (!U.compareAndSwapInt(this, PENDING, c = pending, c+delta));
513     }
514 
515     /**
516      * Sets (atomically) the pending count to the given count only if
517      * it currently holds the given expected value.
518      *
519      * @param expected the expected value
520      * @param count the new value
521      * @return true if successful
522      */
523     public final boolean compareAndSetPendingCount(int expected, int count) {
524         return U.compareAndSwapInt(this, PENDING, expected, count);
525     }
526 
527     /**
528      * If the pending count is nonzero, (atomically) decrements it.
529      *
530      * @return the initial (undecremented) pending count holding on entry
531      * to this method
532      */
533     public final int decrementPendingCountUnlessZero() {
534         int c;
535         do {} while ((c = pending) != 0 &&
536                      !U.compareAndSwapInt(this, PENDING, c, c - 1));
537         return c;
538     }
539 
540     /**
541      * Returns the root of the current computation; i.e., this
542      * task if it has no completer, else its completer's root.
543      *
544      * @return the root of the current computation
545      */
546     public final CountedCompleter<?> getRoot() {
547         CountedCompleter<?> a = this, p;
548         while ((p = a.completer) != null)
549             a = p;
550         return a;
551     }
552 
553     /**
554      * If the pending count is nonzero, decrements the count;
555      * otherwise invokes {@link #onCompletion} and then similarly
556      * tries to complete this task's completer, if one exists,
557      * else marks this task as complete.
558      */
559     public final void tryComplete() {
560         CountedCompleter<?> a = this, s = a;
561         for (int c;;) {
562             if ((c = a.pending) == 0) {
563                 a.onCompletion(s);
564                 if ((a = (s = a).completer) == null) {
565                     s.quietlyComplete();
566                     return;
567                 }
568             }
