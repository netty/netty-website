31  /**
32   * A life cycled buffer of memory, with separate reader and writer offsets.
33   * <p>
34   * A buffer is a logically sequential stretch of memory with a certain capacity, an offset for writing,
35   * and an offset for reading.
36   * Buffers may be {@linkplain CompositeBuffer composed} of multiple {@linkplain #countComponents() components},
37   * where each component is a guaranteed contiguous chunk of memory.
38   *
39   * <h3>Creating a buffer</h3>
40   *
41   * Buffers are created by {@linkplain BufferAllocator allocators}, and their {@code allocate} family of methods.
42   * A number of standard allocators exist, and are available through static methods on the {@code BufferAllocator}
43   * interface.
44   *
45   * <h3>Buffer life cycle</h3>
46   *
47   * The buffer has a life cycle, where it is allocated, used, and deallocated.
48   * When the buffer is initially allocated, a pairing {@link #close()} call will deallocate it.
49   * If a buffer is {@linkplain #send() sent} elsewhere, the {@linkplain #close() close} method on the given instance
50   * will become a no-op.
51   * The buffer can be thought of as a view onto memory, and calling {@link #send()} on the buffer will effectively close
52   * that view, and recreate it upon reception at its destination.
53   *
54   * <h3>Thread-safety</h3>
55   *
56   * Buffers are <strong>not</strong> thread-safe.
57   *
58   * <h3>Accessing data</h3>
59   *
60   * Data access methods fall into two classes:
61   * <ol>
62   *     <li>Access that are based on, and updates, the read or write offset positions.</li>
63   *     <ul><li>These accessor methods are typically called {@code readX} or {@code writeX}.</li></ul>
64   *     <li>Access that take offsets as arguments, and do not update read or write offset positions.</li>
65   *     <ul><li>These accessor methods are typically called {@code getX} or {@code setX}.</li></ul>
66   * </ol>
67   *
68   * A buffer contains two mutable offset positions: one for reading and one for writing.
69   * These positions use <a href="https://en.wikipedia.org/wiki/Zero-based_numbering">zero-based indexing</a>,
70   * such that the first byte of data in the buffer is placed at offset {@code 0},
71   * and the last byte in the buffer is at offset {@link #capacity() capacity - 1}.
72   * The {@link #readerOffset()} is the offset into the buffer from which the next read will take place,
73   * and is initially zero.
74   * The reader offset must always be less than or equal to the {@link #writerOffset()}.
75   * The {@link #writerOffset()} is likewise the offset into the buffer where the next write will take place.
76   * The writer offset is also initially zero, and must be less than or equal to the {@linkplain #capacity() capacity}.
77   * <p>
78   * This carves the buffer into three regions, as demonstrated by this diagram:
79   * <pre>
80   *      +-------------------+------------------+------------------+
81   *      | discardable bytes |  readable bytes  |  writable bytes  |
82   *      |                   |     (CONTENT)    |                  |
83   *      +-------------------+------------------+------------------+
84   *      |                   |                  |                  |
85   *      0      <=     readerOffset  <=   writerOffset    <=    capacity
86   * </pre>
87   *
88   * <h3>Byte Order</h3>
89   *
90   * Buffers are always big endian, and this cannot be changed.
91   * Usages that need to get, set, read, or write, little-endian values will have to flip the byte order of the values
92   * they read and write.
93   *
94   * <h3 name="split">Splitting buffers</h3>
95   *
96   * The {@link #split()} method breaks a buffer into two.
97   * The two buffers will share the underlying memory, but their regions will not overlap, ensuring that the memory is
98   * safely shared between the two.
99   * <p>
100  * Splitting a buffer is useful for when you want to hand over a region of a buffer to some other,
101  * perhaps unknown, piece of code, and relinquish your ownership of that buffer region in the process.
102  * Examples include aggregating messages into an accumulator buffer, and sending messages down the pipeline for
103  * further processing, as split buffer regions, once their data has been received in its entirety.
104  *
105  * If you instead wish to temporarily share a region of a buffer, you will have to pass offset and length along with the
106  * buffer, or you will have to make a copy of the region.
107  *
108  * <h3>Buffers as constants</h3>
109  *
110  * Sometimes, the same bit of data will be processed or transmitted over and over again. In such cases, it can be
111  * tempting to allocate and fill a buffer once, and then reuse it.
112  * Such reuse must be done carefully, however, to avoid a number of bugs.
113  * The {@link BufferAllocator} has a {@link BufferAllocator#constBufferSupplier(byte[])} method that solves this, and
114  * prevents these bugs from occurring.
115  */

229     /**
230      * Set an upper limit to the implicit capacity growth. Buffer {@code write*} methods may implicitly grow the buffer

232      * buffer capacity does not automatically grow beyond the given limit. When the limit is reached, and there is no

244      * @return This buffer instance.

246      *                                   {@linkplain #capacity() capacity} is already greater than the given limit.

399     /**
400      * Writes into this buffer, all the readable bytes from the given buffer.
401      * This updates the {@linkplain #writerOffset() write offset} of this buffer, and the
402      * {@linkplain #readerOffset() reader offset} of the given buffer.
403      *
404      * @param source The buffer to read from.
405      * @return This buffer.
406      * @throws NullPointerException If the source buffer is {@code null}.
407      */
408     default Buffer writeBytes(Buffer source) {
409         int size = source.readableBytes();
410         if (writableBytes() < size) {
411             ensureWritable(size, 1, false);
412         }
413         int woff = writerOffset();
414         source.copyInto(source.readerOffset(), this, woff, size);
415         source.skipReadable(size);
416         skipWritable(size);
417         return this;
418     }

425      * @return This buffer.

428         return writeBytes(source, 0, source.length);

431     /**
432      * Writes into this buffer, the given number of bytes from the byte array.
433      * This updates the {@linkplain #writerOffset() write offset} of this buffer by the length argument.
434      *
435      * @param source The byte array to read from.
436      * @param srcPos Position in the {@code source} from where bytes should be written to this buffer.
437      * @param length The number of bytes to copy.
438      * @return This buffer.
439      */
440     default Buffer writeBytes(byte[] source, int srcPos, int length) {
441         if (source.length < srcPos + length) {
442             throw new ArrayIndexOutOfBoundsException();
443         }
444         ensureWritable(length, 1, false);
445         int woff = writerOffset();
446         for (int i = 0; i < length; i++) {
447             setByte(woff + i, source[srcPos + i]);
448         }
449         skipWritable(length);
450         return this;
451     }

453     /**
454      * Writes into this buffer from the source {@link ByteBuffer}.
455      * This updates the {@linkplain #writerOffset() write offset} of this buffer and also the position of
456      * the source {@link ByteBuffer}.
457      * <p>
458      * Note: the behaviour is undefined if the given {@link ByteBuffer} is an alias for the memory in this buffer.
459      *
460      * @param source The {@link ByteBuffer} to read from.
461      * @return This buffer.
462      */
463     default Buffer writeBytes(ByteBuffer source) {
464         if (source.hasArray()) {
465             writeBytes(source.array(), source.arrayOffset() + source.position(), source.remaining());
466             source.position(source.limit());
467         } else {
468             int woff = writerOffset();
469             int length = source.remaining();
470             writerOffset(woff + source.remaining());
471             // Try to reduce bounds-checking by using long and int when possible.
472             boolean needReverse = source.order() != ByteOrder.BIG_ENDIAN;
473             for (; length >= Long.BYTES; length -= Long.BYTES, woff += Long.BYTES) {
474                 setLong(woff, needReverse ? Long.reverseBytes(source.getLong()) : source.getLong());
475             }
476             for (; length >= Integer.BYTES; length -= Integer.BYTES, woff += Integer.BYTES) {
477                 setInt(woff, needReverse ? Integer.reverseBytes(source.getInt()) : source.getInt());
478             }
479             for (; length > 0; length--, woff++) {
480                 setByte(woff, source.get());
481             }
482         }
483         return this;
484     }

487      * Read from this buffer, into the destination {@link ByteBuffer}
488      * This updates the {@linkplain #readerOffset() read offset} of this buffer and also the position of
489      * the destination {@link ByteBuffer}.

491      * Note: the behaviour is undefined if the given {@link ByteBuffer} is an alias for the memory in this buffer.

517         return this;

564     /**
565      * Opens a cursor to iterate the readable bytes of this buffer. The {@linkplain #readerOffset() reader offset} and
566      * {@linkplain #writerOffset() writer offset} are not modified by the cursor.
567      * <p>
568      * Care should be taken to ensure that the buffer's lifetime extends beyond the cursor and the iteration, and that
569      * the {@linkplain #readerOffset() reader offset} and {@linkplain #writerOffset() writer offset} are not modified
570      * while the iteration takes place. Otherwise, unpredictable behaviour might result.
571      *

792     /**
793      * Splits the buffer into two, at {@code length} number of bytes from the current
794      * {@linkplain #readerOffset()} reader offset} position.
795      * <p>
796      * The region of this buffer that contain the previously read and readable bytes till the
797      * {@code readerOffset() + length} position, will be captured and returned in a new buffer,
798      * that will hold its own ownership of that region.
799      * This allows the returned buffer to be independently {@linkplain #send() sent} to other threads.
800      * <p>
801      * The returned buffer will change its {@link #readerOffset()} to {@code readerOffset() + length}, and have its
802      * {@link #writerOffset()} and {@link #capacity()} both set to the {@code readerOffset() + length} position.
803      * <p>
804      * The memory region in the returned buffer will become inaccessible through this buffer. This buffer will have its
805      * capacity reduced by the capacity of the returned buffer, read offset will become zero and relative position of
806      * write offset will be preserved from the provided {@code readerOffset() + length} position,
807      * even though their position in memory remain unchanged.
808      * <p>
809      * Effectively, the following transformation takes place:
810      * <pre>{@code
811      *         This buffer, where offset = readerOffset() + length:
812      *          +------------------------------------------+
813      *         0|   |r/o    offset        |w/o             |cap
814      *          +---+---------+-----------+----------------+
815      *         /   /         / \          \                \
816      *        /   /         /   \          \                \
817      *       /   /         /     \          \                \
818      *      /   /         /       \          \                \
819      *     /   /         /         \          \                \
820      *    +---+---------+           +----------+----------------+
821      *    |   |r/o      |w/o & cap  |r/o      w/o               |cap
822      *    +---+---------+           +---------------------------+
823      *    Returned buffer.                   This buffer.
824      * }</pre>

826      * underlying memory. This means that shared underlying memory will not be deallocated or returned to a pool, until
827      * all the split parts have been closed.

829      * Composite buffers have it a little easier, in that at most only one of the constituent buffers will actually be
830      * split. If the split point lands perfectly between two constituent buffers, then a composite buffer can
831      * simply split its internal array in two.
832      * <p>
833      * Split buffers support all operations that normal buffers do, including {@link #ensureWritable(int)}.
834      * <p>
835      * See the <a href="#split">Splitting buffers</a> section for details.
836      *
837      * @return A new buffer with independent and exclusive ownership over the previously read and readable bytes from
838      * this buffer.

1080     /**
1081      * Create a {@linkplain ComponentIterator component iterator} for all readable components in this buffer.
1082      * <p>
1083      * Unlike the {@link #forEachReadable(int, ReadableComponentProcessor)} method, this API permits <em>external</em>
1084      * iteration of the components, while at the same time protecting the life-cycle of the buffer.
1085      * <p>
1086      * The typical code pattern for using this API looks like the following:
1087      * <pre>{@code
1088      *      try (var iteration = buffer.forEachReadable()) {
1089      *          for (var c = iteration.first(); c != null; c = c.next()) {
1090      *              ByteBuffer componentBuffer = c.readableBuffer();
1091      *              // ...
1092      *          }
1093      *      }
1094      * }</pre>
1095      * Note the use of the {@code var} keyword for local variables, which are required for correctly expressing the
1096      * generic types used in the iteration.
1097      * Following this code pattern will ensure that the components, and their parent buffer, will be correctly
1098      * life-cycled.
1099      * <p>
1100      * <strong>Note</strong> that the {@link ReadableComponent} instances exposed by the iterator could be reused for
1101      * multiple calls, so the data must be extracted from the component in the context of the iteration.
1102      * <p>
1103      * The {@link ByteBuffer} instances obtained from the component, share lifetime with that internal component.
1104      * This means they can be accessed as long as the internal memory store remain unchanged. Methods that may cause
1105      * such changes are {@link #split(int)}, {@link #split()}, {@link #readSplit(int)}, {@link #writeSplit(int)},
1106      * {@link #compact()}, {@link #ensureWritable(int)}, {@link #ensureWritable(int, int, boolean)},
1107      * and {@link #send()}.
1108      * <p>
1109      * The best way to ensure this doesn't cause any trouble, is to use the buffers directly as part of the iteration.
1110      * <p>
1111      * <strong>Note</strong> that the arrays, memory addresses, and byte buffers exposed as components by this method,
1112      * should not be used for changing the buffer contents. Doing so may cause undefined behaviour.
1113      * <p>
1114      * Changes to position and limit of the byte buffers exposed via the processed components, are not reflected back to
1115      * this buffer instance.
1116      *
1117      * @return A component iterator of {@linkplain ReadableComponent readable components}.
1118      * @param <T> An intersection type that presents both the {@link ReadableComponent} interface,
1119      *          <em>and</em> the ability to progress the iteration via the {@link Next#next()} method.
1120      */
