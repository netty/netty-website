29  /**
30   * Handles an I/O event or intercepts an I/O operation, and forwards it to its next handler in
31   * its {@link ChannelPipeline}.
32   *
33   * <h3>Extend {@link ChannelHandlerAdapter} instead</h3>
34   * <p>
35   * Because this interface has many methods to implement, you might want to extend {@link ChannelHandlerAdapter}
36   * instead.
37   * </p>
38   *
39   * <h3>The context object</h3>
40   * <p>
41   * A {@link ChannelHandler} is provided with a {@link ChannelHandlerContext}
42   * object.  A {@link ChannelHandler} is supposed to interact with the
43   * {@link ChannelPipeline} it belongs to via a context object.  Using the
44   * context object, the {@link ChannelHandler} can pass events upstream or
45   * downstream, modify the pipeline dynamically, or store the information
46   * (attachment) which is specific to the handler.
47   *
48   * <h3>State management</h3>
49   *
50   * A {@link ChannelHandler} often needs to store some stateful information.
51   * The simplest and recommended approach is to use member variables:
52   * <pre>
53   * public interface Message {
54   *     // your methods here
55   * }
56   *
57   * public class DataServerHandler extends {@link SimpleChannelInboundHandler}&lt;Message&gt; {
58   *
59   *     <b>private boolean loggedIn;</b>
60   *
61   *     {@code @Override}
62   *     public void channelRead({@link ChannelHandlerContext} ctx, Message message) {
63   *         {@link Channel} ch = e.getChannel();
64   *         if (message instanceof LoginMessage) {
65   *             authenticate((LoginMessage) message);
66   *             <b>loggedIn = true;</b>
67   *         } else (message instanceof GetDataMessage) {
68   *             if (<b>loggedIn</b>) {
69   *                 ch.write(fetchSecret((GetDataMessage) message));
70   *             } else {
71   *                 fail();
72   *             }
73   *         }
74   *     }
75   *     ...
76   * }
77   * </pre>
78   * Because the handler instance has a state variable which is dedicated to
79   * one connection, you have to create a new handler instance for each new
80   * channel to avoid a race condition where a unauthenticated client can get
81   * the confidential information:
82   * <pre>
83   * // Create a new handler instance per channel.
84   * // See {@link ChannelInitializer#initChannel(Channel)}.
85   * public class DataServerInitializer extends {@link ChannelInitializer}&lt{@link Channel}&gt {
86   *     {@code @Override}
87   *     public void initChannel({@link Channel} channel) {
88   *         channel.pipeline().addLast("handler", <b>new DataServerHandler()</b>);
89   *     }
90   * }
91   *
92   * </pre>
93   *
94   * <h4>Using an attachment</h4>
95   *
96   * Although it's recommended to use member variables to store the state of a
97   * handler, for some reason you might not want to create many handler instances.
98   * In such a case, you can use an <em>attachment</em> which is provided by
99   * {@link ChannelHandlerContext}:
100  * <pre>
101  * public interface Message {
102  *     // your methods here
103  * }
104  *
105  * {@code @Sharable}
106  * public class DataServerHandler extends {@link SimpleChannelInboundHandler}&lt;Message&gt; {
107  *   private final {@link AttributeKey}&lt{@link Boolean}&gt auth =
108  *           new {@link AttributeKey}&lt{@link Boolean}&gt("auth");
109  *
110  *   // This handler will receive a sequence of increasing integers starting
111  *   // from 1.
112  *   {@code @Override}
113  *   public void channelRead({@link ChannelHandlerContext} ctx, {@link Integer} integer) {
114  *     {@link Attribute}&lt{@link Boolean}&gt attr = ctx.getAttr(auth);
115  *
116  *     {@code @Override}
117  *     public void channelRead({@link ChannelHandlerContext} ctx, Message message) {
118  *         {@link Channel} ch = ctx.channel();
119  *         if (message instanceof LoginMessage) {
120  *             authenticate((LoginMessage) o);
121  *             <b>attr.set(true)</b>;
122  *         } else (message instanceof GetDataMessage) {
123  *             if (<b>Boolean.TRUE.equals(attr.get())</b>) {
124  *                 ch.write(fetchSecret((GetDataMessage) o));
125  *             } else {
126  *                 fail();
127  *             }
128  *         }
129  *     }
130  *     ...
131  * }
132  * </pre>
133  * Now that the state of the handler is stored as an attachment, you can add the
134  * same handler instance to different pipelines:
135  * <pre>
136  * public class DataServerInitializer extends {@link ChannelInitializer}&lt{@link Channel}&gt {
137  *
138  *     private static final DataServerHandler <b>SHARED</b> = new DataServerHandler();
139  *
140  *     {@code @Override}
141  *     public void initChannel({@link Channel} channel) {
142  *         channel.pipeline().addLast("handler", <b>SHARED</b>);
143  *     }
144  * }
145  * </pre>
146  *
147  *
148  * <h4>The {@code @Sharable} annotation</h4>
149  * <p>
150  * In the examples above which used an attachment,
151  * you might have noticed the {@code @Sharable} annotation.
152  * <p>
153  * If a {@link ChannelHandler} is annotated with the {@code @Sharable}
154  * annotation, it means you can create an instance of the handler just once and
155  * add it to one or more {@link ChannelPipeline}s multiple times without
156  * a race condition.
157  * <p>
158  * If this annotation is not specified, you have to create a new handler
159  * instance every time you add it to a pipeline because it has unshared state
160  * such as member variables.
161  * <p>
162  * This annotation is provided for documentation purpose, just like
163  * <a href="http://www.javaconcurrencyinpractice.com/annotations/doc/">the JCIP annotations</a>.
164  *
165  * <h3>Additional resources worth reading</h3>
166  * <p>
167  * Please refer to the {@link ChannelHandler}, and
168  * {@link ChannelPipeline} to find out more about inbound and outbound operations,
169  * what fundamental differences they have, how they flow in a  pipeline,  and how to handle
170  * the operation in your application.
171  */

313     /**
314      * Indicates that the same instance of the annotated {@link ChannelHandler}
315      * can be added to one or more {@link ChannelPipeline}s multiple times
316      * without a race condition.
317      * <p>
318      * If this annotation is not specified, you have to create a new handler
319      * instance every time you add it to a pipeline because it has unshared
320      * state such as member variables.
321      * <p>
322      * This annotation is provided for documentation purpose, just like
323      * <a href="http://www.javaconcurrencyinpractice.com/annotations/doc/">the JCIP annotations</a>.
324      */
325     @Inherited
326     @Documented
327     @Target(ElementType.TYPE)
328     @Retention(RetentionPolicy.RUNTIME)
329     @interface Sharable {
330         // no value
331     }
332 
333     /**
334      * Indicates that the annotated event handler method in {@link ChannelHandler} will not be invoked by
335      * {@link ChannelPipeline}.  This annotation is only useful when your handler method implementation
336      * only passes the event through to the next handler, like the following:
337      *
338      * <pre>
339      * {@code @Skip}
340      * {@code @Override}
341      * public void channelActive({@link ChannelHandlerContext} ctx) {
342      *     ctx.fireChannelActive(); // do nothing but passing through to the next handler
343      * }
344      * </pre>
345      *
346      * {@link #handlerAdded(ChannelHandlerContext)} and {@link #handlerRemoved(ChannelHandlerContext)} are not able to
347      * pass the event through to the next handler, so they must do nothing when annotated.
348      *
349      * <pre>
350      * {@code @Skip}
351      * {@code @Override}
352      * public void handlerAdded({@link ChannelHandlerContext} ctx) {
353      *     // do nothing
354      * }
355      * </pre>
356      *
357      * <p>
358      * Note that this annotation is not {@linkplain Inherited inherited}.  If you override a method annotated with
359      * {@link Skip}, it will not be skipped anymore.  Similarly, you can override a method not annotated with
360      * {@link Skip} and simply pass the event through to the next handler, which reverses the behavior of the
361      * supertype.
362      * </p>
363      */
364     @Target(ElementType.METHOD)
365     @Retention(RetentionPolicy.RUNTIME)
366     @interface Skip {
367         // no value
368     }
